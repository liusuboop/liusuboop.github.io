markdown 弊端

markdown 可以说是为程序员开发的一种轻量型标记语言。可以用来写博客，写一下小型文档，对代码排版很友好。尽管word等办公软件很方便，但是对于代码文本支持很不好，写出来很丑。

缺陷

最大的所谓“缺陷”，是缺少一个随时间跟进的官方标准。公认的“官方标准”在这儿，最后的更新时间是2004年12月17日。

十五年过去了，Markdown的用法已经被各家扩展折腾到妈都不认识了，PHP Markdown Extra，GitHub Flavored Markdown，CommonMark，MultiMarkdown，Kramdown……可是扩展来扩展去，没有什么组织或者个人出面，把众多的扩展语法整合归纳，形成一套新的标准。

扩展多了，就给许多人一种错觉，认为Markdown什么都能干，恨不得用它来取代Word/LaTeX。问题在于：

  • 第一，因为没有新的标准来规范各种实现，各家Markdown扩展之间缺乏可移植性，某些独家支持的写法换套工具就崩。
  • 第二，Word和LaTeX能让用户同时操作内容和格式，而Markdown采用的是典型的内容与格式分离的做法，内容一端用Markdown的标记，格式一端交给HTML、CSS和JS，两者的耦合度极低。要求Markdown做到一些格式上的效果（如文本对齐、图片环绕等）是不切实际的，只能打破内容和格式的低耦合，深入底层用DIV+CSS的“脏”技巧来实现。这玩意儿的学习成本事实上不亚于Word或LaTeX，会让人陷入无尽的折腾中不能自拔（有幸拔出来的话，你就可以去做做前端了）。

设计理念

比如政府、机关很多的文字工作最终必须落实到纸张打印上。那么一这点markdown的确是非常非常不好用。==因为markdown的设计理念本来就是为了电子设备自上而下的阅读的==，从一开始就没考虑过什么分页的问题。所以如果是这种情况，那么完全可以弃用markdown了。

鬼裁

我也是个老鬼才了，通过将图片切片的方式导出成 pdf，这样避免了因为图片造成的大面积白色的问题。（当然也可以自己写一个 markdwon 的模板引擎方便导出成 pdf 形式，从设想上看这种应该更好一些）。

应用

markdown 主要应用于两个地方，一个是 readme，一个是写网页文章（尤其是带代码的）。

合理认识

对于专业的文字工作者来说，工作的主要对象是文字、目的也是文字。文字工作者除了要考虑文字本身的信息性，还必须考虑其它因素，比如文字的颜色、字体、排版带给受众的感受等。

而对于大部分社畜来说，文字只是承载信息用的载体，既不是我们工作的主要对象，也不是最终目的。这也符合了Markdown的核心特征之一，通过尽可能剥离文字的不必要的属性，将文字还原为信息本身。

进一步拓展来说。Markdown不仅在尽可能的将文字还原为信息本身，还能协助你更好的组织信息（而非文字），这也是我觉得真香的原因。
